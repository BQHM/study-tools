## 1. Java 集合

### 1.1 HashMap 底层原理是什么？

问题：

HashMap 底层原理是什么？为什么 JDK 8 后会引入红黑树？

一句话答案：

HashMap 底层是数组 + 链表 + 红黑树，通过 hash 定位数组桶，冲突时挂链表，冲突严重时转红黑树来降低查询复杂度。

展开回答：

HashMap 根据 key 的 hash 值计算数组下标。没有冲突时，元素直接放到对应桶中；发生冲突时，多个节点会以链表形式挂在同一个桶后面。JDK 8 之后，当链表长度达到阈值并且数组容量足够大时，链表会转换成红黑树，避免极端哈希冲突下查询退化成 O(n)。

默认负载因子是 0.75，表示容量和空间利用率之间的折中。元素数量超过阈值后会扩容，扩容通常变成原来的 2 倍。

项目关联：

项目里常用 `Map` 做临时数据聚合、枚举映射、缓存结果组装。面试时可以关联 `java-core-lab` 中的 HashMap 扩容、冲突和红黑树 Demo。

常见追问：

- 为什么容量是 2 的幂？
- HashMap 为什么线程不安全？
- 链表转红黑树的条件是什么？
- put 一个元素的大致流程是什么？

容易踩坑：

- 只说“数组 + 链表”，忘了 JDK 8 后有红黑树。
- 说 HashMap 是线程安全的。
- 只背结论，不知道扩容会重新分布元素。

自测标准：

能画出数组桶、链表、红黑树结构，并能解释一次 put 的过程。

### 1.2 HashMap 为什么线程不安全？

一句话答案：

HashMap 没有同步控制，并发 put、扩容、修改链表或树节点时可能出现数据覆盖、丢失、结构异常和 size 不准确。

展开回答：

HashMap 的 put 操作不是原子的。多个线程同时计算桶位置、插入节点、更新 size 或触发扩容时，彼此之间没有可见性和互斥保证。比如两个线程同时向同一个桶插入数据，可能互相覆盖；并发扩容时也可能出现数据丢失或结构异常。

项目关联：

如果需要并发读写共享 Map，应使用 `ConcurrentHashMap`，或者在局部方法内使用 HashMap，避免跨线程共享。

常见追问：

- ConcurrentHashMap 如何保证线程安全？
- Collections.synchronizedMap 和 ConcurrentHashMap 有什么区别？

容易踩坑：

不要把“JDK 8 解决了死循环问题”理解成 HashMap 就线程安全了。

## 2. Java 并发

### 2.1 ConcurrentHashMap 如何保证线程安全？

一句话答案：

JDK 8 的 ConcurrentHashMap 主要通过 CAS、volatile 和桶级 synchronized 保证并发安全，读多写少场景性能较好。

展开回答：

JDK 8 中，ConcurrentHashMap 不再使用 Segment 分段锁，而是把锁粒度降低到桶级别。数组节点使用 volatile 保证可见性。插入时如果桶为空，先用 CAS 尝试写入；如果桶不为空，再对桶头节点加 synchronized，保证同一个桶内修改安全。

读操作大多数情况下不加锁，依赖 volatile 读取到最新可见数据，所以并发读性能较好。

项目关联：

在本地缓存、运行时状态表、统计计数临时聚合中，如果存在多线程读写，可以考虑 ConcurrentHashMap。

常见追问：

- JDK 7 和 JDK 8 的实现区别？
- size 怎么统计？
- 为什么 key 和 value 不允许为 null？

容易踩坑：

ConcurrentHashMap 保证单个操作线程安全，但不能自动保证“先判断再操作”这种复合逻辑的原子性。

### 2.2 volatile 解决什么问题？

一句话答案：

volatile 保证可见性和禁止指令重排序，但不保证复合操作的原子性。

展开回答：

一个线程修改 volatile 变量后，其他线程能及时看到最新值。volatile 还会在读写前后插入内存屏障，限制指令重排序。因此它适合做状态标记、开关变量，也常用于双重检查锁单例中防止对象初始化重排序。

但是 `count++` 包含读取、加一、写回三个步骤，volatile 不能把这三个步骤合成一个原子操作。

项目关联：

可以用 volatile 表示后台任务是否继续运行，但计数器要用 AtomicInteger、LongAdder 或加锁。

常见追问：

- volatile 和 synchronized 区别？
- DCL 单例为什么要加 volatile？
- volatile 能不能保证线程安全？

容易踩坑：

不要说 volatile 可以替代锁。

### 2.3 ThreadLocal 原理和风险？

一句话答案：

ThreadLocal 给每个线程保存一份变量副本，适合存用户上下文，但在线程池中必须及时 remove，避免脏数据和内存泄漏。

展开回答：

每个线程内部都有一个 ThreadLocalMap，ThreadLocal 对象作为 key，业务对象作为 value。不同线程访问同一个 ThreadLocal 时，实际读写的是自己线程里的 value。

在线程池中线程会被复用，如果请求结束后没有清理 ThreadLocal，下一个请求可能读到上一个请求的数据。ThreadLocalMap 的 key 是弱引用，但 value 仍可能被线程引用，长期不清理会增加内存泄漏风险。

项目关联：

登录鉴权后可以把当前用户 ID 放到 UserContext，请求结束在过滤器 finally 中清理。

常见追问：

- ThreadLocalMap 的 key 为什么是弱引用？
- 为什么弱引用还会内存泄漏？
- InheritableThreadLocal 是什么？

容易踩坑：

只写 `set` 不写 `remove`，在线程池 Web 服务里很危险。

### 2.4 线程池执行流程是什么？

一句话答案：

线程池提交任务后，先用核心线程，核心满了进队列，队列满了再创建非核心线程，达到最大线程数后触发拒绝策略。

展开回答：

ThreadPoolExecutor 的核心参数包括核心线程数、最大线程数、阻塞队列、空闲线程存活时间、线程工厂和拒绝策略。任务提交后，如果运行线程数小于核心线程数，就创建核心线程执行；否则尝试放入队列；队列满了并且线程数小于最大线程数，就创建非核心线程；再满就执行拒绝策略。

项目关联：

AI 分析、文件解析、报告生成这类耗时任务，可以先用线程池异步处理。后续规模更大时再引入 MQ。

常见追问：

- 线程池参数怎么设置？
- 有哪些拒绝策略？
- 线程池满了怎么排查？

容易踩坑：

不要直接使用 `Executors.newFixedThreadPool` 而不关心队列大小。无界队列可能导致内存堆积。

## 3. JVM

### 3.1 CPU 飙高怎么排查？

一句话答案：

先找到高 CPU 进程和线程，再用线程栈定位具体代码，最后结合日志和业务场景判断原因。

展开回答：

Linux 常见流程是：用 `top` 找到 Java 进程，用 `top -Hp <pid>` 找到高 CPU 线程，把线程 ID 转成 16 进制，再用 `jstack <pid>` 查找对应 nid 的线程栈。常见原因包括死循环、频繁 GC、锁竞争、正则回溯、序列化或加解密热点等。

本地 Windows 学习时可以使用 VisualVM、JConsole 或 Arthas 辅助观察。

项目关联：

`java-core-lab` 中的 HighCpuDemo 可以作为排查练习素材。

常见追问：

- 如果是频繁 GC 导致 CPU 高怎么办？
- 如果线程栈一直变化怎么看？
- Arthas 用过哪些命令？

容易踩坑：

不要只说“看日志”，CPU 飙高通常需要线程维度定位。

## 4. Spring

### 4.1 Spring 事务为什么会失效？

一句话答案：

Spring 声明式事务依赖代理增强，绕过代理、异常被吞、方法不可代理或回滚规则不匹配时，事务就可能失效。

展开回答：

常见失效场景包括：同类内部方法自调用、方法不是 public、异常被 catch 后没有抛出、默认只回滚 RuntimeException、类没有交给 Spring 管理、数据库引擎不支持事务等。

本质原因是 `@Transactional` 通过 Spring AOP 代理实现，只有从代理对象进入方法时事务增强才会生效。

项目关联：

下单、库存扣减、文件元数据保存、订单状态流转都需要事务边界。比如订单创建和库存流水写入必须在同一事务中保证一致性。

常见追问：

- 同类自调用怎么解决？
- checked exception 怎么回滚？
- 事务传播行为有哪些？

容易踩坑：

在事务方法里 catch 异常后只打印日志不抛出，会导致事务提交。

### 4.2 Spring Security + JWT 登录态怎么设计？

一句话答案：

登录成功后签发 JWT，后续请求携带 Token，服务端过滤器校验签名和过期时间，再构建认证上下文；退出登录可用 Redis 黑名单实现主动失效。

展开回答：

用户登录时校验账号密码，成功后生成带用户 ID、过期时间和签名的 JWT。前端请求时在 Authorization Header 中携带 Bearer Token。后端过滤器解析 Token，如果合法就把用户身份放入 SecurityContext 或自定义上下文。因为 JWT 默认无状态，服务端无法天然主动失效，所以退出登录、改密码、封禁账号等场景可以引入 Redis 黑名单。

项目关联：

FixLedger 的登录认证可以对应 Spring Security、JWT、Redis 黑名单、ThreadLocal 用户上下文这些问题。

常见追问：

- JWT 被盗怎么办？
- Refresh Token 怎么设计？
- JWT 放 localStorage 还是 Cookie？

容易踩坑：

不要说 JWT 一定比 Session 安全。JWT 只是无状态，安全性取决于存储、传输、过期时间和失效机制。

## 5. MySQL

### 5.1 索引为什么能加速查询？

一句话答案：

InnoDB 常用 B+Tree 索引，通过有序、多叉、低高度的树结构减少磁盘 IO，并支持范围查询。

展开回答：

B+Tree 的非叶子节点只存键和指针，叶子节点存数据或主键值，因此单个节点可以容纳更多 key，树高更低，查询时磁盘 IO 更少。叶子节点之间有链表，天然适合范围查询。

InnoDB 主键索引是聚簇索引，叶子节点保存整行数据；二级索引叶子节点保存主键值，查到主键后可能需要回表。

项目关联：

设备列表、简历列表、订单列表这类分页查询，需要根据查询条件设计联合索引。

常见追问：

- 回表是什么？
- 覆盖索引是什么？
- 联合索引最左前缀是什么？
- 索引失效有哪些场景？

容易踩坑：

不是加了索引一定会走索引。优化器会根据区分度、数据量、成本决定执行计划。

### 5.2 MVCC 是什么？

一句话答案：

MVCC 是多版本并发控制，通过 undo log 和 Read View 让读写尽量不互相阻塞，提高并发性能。

展开回答：

InnoDB 为记录维护多个版本。快照读时，事务会根据 Read View 判断哪个版本对自己可见。如果某行最新版本不可见，就沿着 undo log 找旧版本。这样普通查询可以读到事务开始时或语句开始时的一致性视图，而不必阻塞正在写入的事务。

项目关联：

在订单、库存、状态流转场景中，需要区分快照读和当前读。普通 select 是快照读，`select ... for update` 属于当前读，会读取最新数据并加锁。

常见追问：

- Read View 里有什么？
- RC 和 RR 下 Read View 创建时机有什么区别？
- undo log 和 redo log 区别？

容易踩坑：

MVCC 不是万能的。涉及库存扣减、状态修改时，仍需要当前读、行锁或乐观锁。

## 6. Redis

### 6.1 缓存穿透、击穿、雪崩怎么回答？

一句话答案：

穿透是查不存在数据，击穿是热点 Key 过期，雪崩是大量 Key 同时失效或 Redis 故障，分别用空值/布隆过滤器、互斥锁/逻辑过期、随机过期/限流降级解决。

展开回答：

缓存穿透会让不存在的数据每次都打到数据库，可以缓存空值、做参数校验或使用布隆过滤器。缓存击穿发生在热点 Key 过期瞬间，大量请求同时查数据库，可以使用互斥锁、逻辑过期或热点 Key 不过期。缓存雪崩是大量缓存同时失效或 Redis 整体不可用，需要过期时间随机化、Redis 高可用、限流降级和本地缓存兜底。

项目关联：

首页看板、设备详情、活动库存都可能使用缓存。核心是缓存失败不能把数据库打垮。

常见追问：

- 布隆过滤器有什么缺点？
- 逻辑过期怎么实现？
- Redis 挂了怎么办？

容易踩坑：

不要只背三个名词，要能说出不同问题的触发场景。

### 6.2 Redis Lua 为什么能防止库存超卖？

一句话答案：

Lua 脚本在 Redis 中原子执行，可以把判断库存、判断是否买过、扣减库存、记录购买标记放在一个不可插队的操作里。

展开回答：

如果先查库存再扣库存，中间可能被其他请求插入，导致并发超卖。Lua 脚本可以把多个 Redis 命令组合成一个原子脚本。秒杀场景下，脚本先判断用户是否买过，再判断库存是否大于 0，然后扣减库存并记录用户购买标记。

项目关联：

未来高并发下单项目中，库存预热到 Redis 后，抢购接口先执行 Lua，成功后再发送 MQ 异步创建订单。

常见追问：

- Redis 扣成功但 MQ 发送失败怎么办？
- Lua 脚本执行太久有什么问题？
- Redis 和 MySQL 库存不一致怎么办？

容易踩坑：

Lua 只能保证 Redis 内部操作原子，不能自动保证 Redis、MQ、MySQL 跨系统事务一致。

## 7. MQ

### 7.1 为什么要引入 MQ？

一句话答案：

MQ 主要用于异步削峰、系统解耦、失败重试和最终一致性处理。

展开回答：

在高并发下单场景中，如果请求直接写数据库，数据库可能被瞬时流量打满。引入 MQ 后，接口只完成前置校验和 Redis 扣减，然后把下单请求发送到队列，由消费者按能力异步落库。这样可以削峰，也能把下单、通知、积分等后续动作解耦。

项目关联：

interview-assistant 可以先用线程池做异步分析，后续再演进到 MQ；高并发下单项目则适合用 MQ 做异步下单。

常见追问：

- MQ 消息丢了怎么办？
- MQ 重复消费怎么办？
- 消息积压怎么办？
- 死信队列怎么处理？

容易踩坑：

MQ 引入后系统不是更简单，而是多了最终一致性、重复消费和补偿问题。

### 7.2 MQ 重复消费怎么处理？

一句话答案：

消费者必须做幂等，常用唯一索引、业务状态机、去重表或 Redis 去重 Key。

展开回答：

MQ 为保证可靠性，可能出现重复投递，所以消费者不能假设一条消息只处理一次。比如秒杀下单时，可以在订单表上建立 `activity_id + user_id` 唯一索引。即使同一消息重复消费，第二次插入也会失败，不会生成重复订单。支付回调可以用支付流水号做唯一约束，订单取消可以通过状态机保证只取消一次。

项目关联：

高并发下单项目一定要讲订单幂等，否则会被追问打穿。

常见追问：

- 唯一索引异常算不算消费成功？
- Redis 去重 Key 过期怎么办？
- 状态机如何防止重复回滚库存？

容易踩坑：

不要只说“用 Redis 记录一下”，要说明 key 的生命周期、失败时机和兜底方案。

## 8. AI 工程化

### 8.1 RAG 是什么？

一句话答案：

RAG 是检索增强生成，先从知识库检索相关内容，再把检索结果交给大模型回答，用来减少幻觉并引入私有知识。

展开回答：

RAG 的基本流程是：文档上传后解析文本，按规则切分成片段，调用 Embedding 模型转成向量，存入向量数据库。用户提问时，把问题也转成向量，在向量库中检索相似片段，再把片段和问题一起交给大模型生成答案。

项目关联：

导师的参考项目里有知识库和 RAG，这只能作为学习和功能规划参考，不能作为个人项目经历来讲。interview-assistant 后续可以做一个属于自己的简化版：上传面试资料，支持基于资料问答。

常见追问：

- 文档怎么切分？
- TopK 和相似度阈值怎么选？
- 多用户知识库怎么做权限隔离？
- 检索不到内容怎么办？

容易踩坑：

RAG 不能彻底消除幻觉，只是降低幻觉概率。最终仍需要引用来源、置信度和兜底策略。

### 8.2 AI 幻觉怎么处理？

一句话答案：

通过限制上下文、RAG 引用来源、结构化输出、业务规则校验、低置信度兜底和人工确认来降低幻觉影响。

展开回答：

AI 生成内容不能直接当事实使用。业务系统中应明确告诉模型只能基于给定上下文回答；RAG 场景要返回引用来源；结构化输出要经过 JSON 解析和字段校验；金额、日期、枚举、权限等关键字段必须由后端规则校验。AI 失败或低置信度时，可以返回待确认状态、规则版结果或人工处理入口。

项目关联：

FixLedger 中 AI 只适合做票据信息提取和故障建议，不能让模型直接绕过后端校验写设备、保修、维修核心数据。

常见追问：

- 模型输出 JSON 解析失败怎么办？
- Prompt Injection 怎么处理？
- AI 服务超时怎么办？
- 如何做成本控制？

容易踩坑：

不要说“用了 RAG 就不会胡说”。正确说法是“通过 RAG 和校验降低幻觉风险，并设计失败兜底”。

### 8.3 LLM 接入业务系统要注意什么？

一句话答案：

LLM 接入不是简单调 API，而要考虑 Prompt 管理、超时重试、结构化输出、结果校验、权限隔离、日志审计、降级和成本。

展开回答：

后端接入大模型时，应该把 Prompt 模板化和版本化；调用模型要设置超时、重试和限流；输出尽量要求结构化，解析后再做业务校验；涉及用户私有数据时要做权限隔离和脱敏；模型异常时不能影响核心业务链路；同时记录调用耗时、Token 消耗、失败率，方便排查和成本控制。

项目关联：

interview-assistant 的简历分析可以设计成异步任务：简历先保存，分析状态为待处理，AI 失败时回退到规则版分析。

常见追问：

- AI 接口失败会不会阻塞主流程？
- 怎么防止用户恶意 Prompt？
- 模型切换后结果不一致怎么办？

容易踩坑：

只写“调用 AI 分析简历”太薄，要讲清楚稳定性和工程兜底。

## 9. 高并发下单

### 9.1 如何防止超卖？

一句话答案：

用 Redis Lua 做前置原子扣减，用数据库唯一约束和库存更新条件做最终兜底，再通过补偿任务处理跨系统不一致。

展开回答：

高并发抢购中，库存先从数据库预热到 Redis。用户请求进来后，先执行 Lua 脚本判断库存和是否重复购买，成功后扣减 Redis 库存并记录用户购买标记，再发送 MQ 异步创建订单。数据库落库时仍要通过唯一索引防重复订单，通过库存扣减条件防止持久层超卖。

如果 Redis 扣成功但后续 MQ 或数据库失败，需要有库存回滚、死信处理或对账补偿。

项目关联：

这是未来高并发学习项目的核心链路。

常见追问：

- Redis 扣成功但订单没创建怎么办？
- Redis 和 DB 库存不一致怎么办？
- 用户重复点击怎么处理？
- 订单超时取消怎么保证只回滚一次？

容易踩坑：

只用 Redis 扣库存还不够，数据库层也要有兜底约束。

### 9.2 订单超时取消如何保证幂等？

一句话答案：

通过订单状态机做条件更新，只有 `UNPAID -> CANCELED` 成功的那一次才允许回滚库存。

展开回答：

订单创建后状态为未支付。如果超过支付时间仍未支付，延迟消息或定时任务触发取消。取消时不能直接更新状态和加库存，而是执行条件更新：只有当前状态还是 UNPAID 才能改成 CANCELED。更新成功说明本次取消有效，再回滚库存；如果更新失败，说明订单可能已支付或已取消，不能重复回滚。

项目关联：

高并发下单项目的订单状态机可以设计为：UNPAID、PAID、CANCELED、CLOSED。

常见追问：

- 支付成功和超时取消并发怎么办？
- Redis 库存和 DB 库存都要回滚吗？
- 延迟消息丢了怎么办？

容易踩坑：

不要在未判断订单状态的情况下直接恢复库存，否则重复取消会导致库存越加越多。

## 10. 项目深挖：interview-assistant

### 10.1 interview-assistant 这个项目怎么介绍？

问题：

请介绍一下 `interview-assistant` 项目。

一句话答案：

`interview-assistant` 是一个基于 Spring Boot 和 Spring AI 的智能面试助手后端 MVP，核心链路是简历上传分析和文字模拟面试。

第二条链路是文字模拟面试：用户基于简历创建面试会话，系统根据简历关键词生成规则版题目，用户可以暂存答案或正式提交答案，提交后推进当前题索引，完成后生成规则版报告。

自测标准：

### 10.2 简历上传分析链路怎么讲？

问题：

`interview-assistant` 的简历上传后端链路是什么？

一句话答案：

简历上传链路是文件校验、内容 hash 去重、文本解析、对象存储、数据库落库、AI 分析和规则兜底。

展开回答：

用户上传简历后，Controller 把文件交给 `ResumeUploadService`。Service 先校验文件类型和大小，再用文件内容计算 hash。如果 hash 已存在，就直接返回已有简历记录，避免同一份简历重复入库。没有重复时，系统用 Apache Tika 解析简历文本，解析为空则返回业务错误。随后把原文件上传到 RustFS / S3 兼容对象存储，并把文件名、大小、类型、storageKey、hash 和解析文本保存到 PostgreSQL。

保存基础信息后，系统调用 `ResumeGradingService` 做简历分析。AI 成功时保存结构化分析结果；AI 调用失败或结构化解析失败时，回退到本地规则版评分和建议。

项目关联：

这条链路可以引出文件上传安全、对象存储、数据库唯一约束、AI 兜底和事务边界问题。

常见追问：

- 为什么用内容 hash，而不是文件名去重？
- 如果对象存储成功但数据库保存失败怎么办？
- Tika 解析失败如何处理？
- 上传接口里同步调用 AI 有什么问题？

容易踩坑：

不要只说“上传文件然后调 AI”。面试里要把校验、去重、解析、存储、落库、分析和兜底讲成完整链路。

自测标准：

能画出从 `POST /api/resumes/upload` 到数据库和对象存储的流程图。

### 10.3 AI 简历分析失败怎么办？

问题：

AI 简历分析失败时，系统怎么保证主流程可用？

一句话答案：

AI 分析失败时回退到规则版分析，并把分析状态和错误信息记录下来，避免外部模型故障拖垮上传主链路。

展开回答：

`ResumeGradingService` 优先通过 Spring AI 调用大模型，并要求模型按 DTO 结构返回结果。由于模型可能超时、返回空内容、输出格式不符合要求，代码会捕获异常并走本地规则版分析。规则版分析根据简历长度、章节关键词、技能关键词、表达和项目描述做基础评分，生成 summary、strengths 和 suggestions。

当前实现是同步分析，学习阶段可以接受，但工程上后续更适合拆成异步任务：上传简历只保存文件和文本，分析状态先置为 PENDING，再由后台任务调用 AI，失败时重试并最终标记 FAILED。

项目关联：

这是 `interview-assistant` 最适合讲 AI 工程化的点：AI 不是可靠数据库，必须做结构化输出、解析校验、异常兜底和状态记录。

常见追问：

- 模型输出 JSON 解析失败怎么办？
- AI 服务超时会不会阻塞上传接口？
- 如何防止 AI 结果污染核心业务数据？
- 后续为什么要异步化？

容易踩坑：

不要说“AI 失败就重新调一次”这么薄。要讲超时、结构化解析、规则降级、状态记录和后续异步化。

自测标准：

能说明 AI 分析成功路径、失败路径和未来异步演进方案。

### 10.4 面试会话状态怎么设计？

问题：

`interview-assistant` 的文字模拟面试会话如何推进？

一句话答案：

一场面试用会话表保存题目快照、总题数、当前题索引和状态；暂存答案不推进索引，提交答案才推进，答完后状态变为 COMPLETED。

展开回答：

用户基于简历创建面试时，系统会先检查同一简历是否存在 CREATED 或 IN_PROGRESS 的未完成会话，如果有就复用，避免重复创建多场并行面试。没有未完成会话时，系统根据简历关键词生成规则版题目，并把题目快照保存下来。

`currentQuestionIndex` 表示下一道待答题的索引，而不是最近一道已答题索引。获取当前题时直接按这个索引取题。暂存答案只保存用户输入，不推进索引；正式提交答案时要求按顺序提交，保存答案后把索引加一。如果索引达到题目总数，会话状态改为 COMPLETED，之后不能继续暂存或提交。

项目关联：

这可以关联状态机、幂等、重复提交、历史详情和报告生成。

常见追问：

- 为什么要复用未完成会话？
- 暂存答案和提交答案区别是什么？
- 如果用户重复提交同一题怎么办？
- 提前交卷后还能不能生成报告？

容易踩坑：

不要把 `currentQuestionIndex` 讲成“已答题索引”。它在当前设计里是下一道待答题索引。

自测标准：

能口述 CREATED、IN_PROGRESS、COMPLETED 三个状态的进入条件。

### 10.5 为什么答案要拆成独立答案表？

问题：

为什么不能一直把题目和答案都放在 `questionsJson` 里？

一句话答案：

`questionsJson` 适合 MVP 保存题目快照，但历史详情、单题评分、AI 评估和报告导出都需要独立答案表来表达和查询答案。

展开回答：

早期为了快速跑通主流程，可以把题目列表作为 JSON 快照存在会话表里，这样创建会话和读取题目都比较简单。但当系统需要暂存答案、多次更新同一题答案、查询历史详情、统计已答未答、做单题评分、生成 PDF 报告时，继续把答案塞在 JSON 里会让查询、更新和扩展都变复杂。

独立答案表可以用 `session_id + question_index` 做唯一约束，保证同一场面试同一题只有一条答案。题目快照仍然可以留在 `questionsJson`，答案、得分、反馈、参考答案和关键点则放到答案表。报告和详情再把题目快照和答案表聚合起来。

项目关联：

`interview-assistant` 最新代码已经出现 `InterviewAnswerEntity`，下一步接管重点是确认暂存、提交、历史详情和报告都真正基于答案表聚合，而不是部分逻辑还在读旧 JSON。

常见追问：

- 唯一约束应该怎么设计？
- 暂存和提交如何避免插入重复答案？
- 历史详情如何展示未答题？
- 报告统计应该基于题目表、会话表还是答案表？

容易踩坑：

拆答案表后，只改保存逻辑不够；查询详情和报告生成也要一起改，否则会出现“答案已保存但报告读不到”的问题。

自测标准：

能说明 `questionsJson` 和 `interview_answers` 的职责边界。

### 11.1 如何介绍自己的 AI Agent 工程方向？

如果面试官问“你为什么投 AI Agent 工程师，你的优势是什么”，怎么回答？

### 11.2 Agent 和普通 ChatBot 有什么区别？

问题：

AI Agent 和普通聊天机器人有什么区别？

一句话答案：

普通 ChatBot 主要是回答问题，Agent 还要能理解目标、规划步骤、调用工具、维护记忆，并根据结果继续执行。

展开回答：

普通 ChatBot 更像单轮或多轮问答系统，用户问什么，模型回答什么。Agent 则更强调完成任务：它需要理解用户目标，把目标拆成步骤，决定是否调用工具，读取工具结果，再决定下一步。更完整的 Agent 还会维护短期会话记忆和长期用户记忆。

比如在 `interview-assistant` 里，普通 ChatBot 只能回答“HashMap 是什么”。AI 面试准备 Agent 则应该能读取用户简历和 JD，判断薄弱点，生成学习计划，调用知识库出题，根据用户回答评分，再记录薄弱知识点，用于下一轮训练。

项目关联：

后续可以把 `interview-assistant` 的能力拆成工具：简历分析工具、出题工具、答案评估工具、知识库检索工具和学习计划工具。

常见追问：

- Agent 怎么选择工具？
- 工具调用失败怎么办？
- Agent 的记忆存在哪里？
- Agent 自主规划会不会失控？

容易踩坑：

不要把“接了大模型接口”就说成 Agent。Agent 至少要有任务目标、工具调用或状态推进中的一部分。

自测标准：

能用 `interview-assistant` 举例说明 ChatBot 和 Agent 的差异。
