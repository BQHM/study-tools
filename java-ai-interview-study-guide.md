> 适用目标：Java 后端开发、Java 实习/校招/社招初中级、AI 辅助业务系统开发、项目复盘与简历准备。

## 0. 复习总原则

### 面试回答结构

每个技术点都按这 5 句话组织：

```text
1. 它解决什么问题？
2. 核心原理是什么？
3. 项目里怎么用？
4. 有什么坑？
5. 出问题怎么排查或兜底？
```

举例：Redis 缓存不是只说“提高性能”，而是要说：

```text
高频读接口直接查 MySQL 压力大，所以把热点数据放入 Redis。
读时先查 Redis，未命中再查 DB 并回填。
要处理缓存穿透、击穿、雪崩、数据一致性和热点 Key。
如果缓存异常，核心链路要能降级到 DB 或返回明确错误。
```

| 优先级 | 模块 | 原因 |

## 1. Java 基础高频题

### 1.1 Java 面向对象有哪些特性？

必会回答：

- 封装：隐藏内部实现，通过方法暴露能力。
- 继承：复用父类能力，但容易带来强耦合。
- 多态：父类引用指向子类对象，运行时动态绑定方法。
- 抽象：提取共同能力，用接口或抽象类表达稳定契约。

追问点：

- 接口和抽象类区别？
- Java 为什么不支持类的多继承？
- 重载和重写区别？

项目表达：

```text
文件存储可以抽象成 FileStorageService，具体实现可以是本地存储、MinIO/RustFS、S3。业务模块只依赖接口，不依赖具体存储实现。
```

### 1.2 equals 和 hashCode 有什么关系？

必会回答：

- `equals` 判断两个对象业务上是否相等。
- `hashCode` 用于哈希容器定位桶。
- 两个对象 `equals` 为 `true`，`hashCode` 必须相同。
- 两个对象 `hashCode` 相同，`equals` 不一定为 `true`，可能哈希冲突。

常见坑：

- 重写 `equals` 不重写 `hashCode`，会导致 `HashSet`、`HashMap` 行为异常。
- 实体类用可变字段参与 hash 计算，放进集合后字段变了会找不到。

### 1.3 String、StringBuilder、StringBuffer 区别？

必会回答：

- `String` 不可变，适合少量字符串拼接和常量。
- `StringBuilder` 可变，非线程安全，性能较好，适合单线程拼接。
- `StringBuffer` 可变，方法加锁，线程安全但性能较差。

追问点：

- String 为什么设计成不可变？
- 字符串常量池是什么？
- `new String("abc")` 创建几个对象？

### 1.4 final、finally、finalize 区别？

必会回答：

- `final`：修饰类、方法、变量。类不可继承，方法不可重写，变量不可重新赋值。
- `finally`：异常处理中的收尾代码块，通常用于释放资源。
- `finalize`：对象被 GC 前可能调用的方法，已不推荐使用。

### 1.5 Java 8 到 Java 21 值得关注什么？

必会回答：

- Java 8：Lambda、Stream、Optional、新时间 API。
- Java 11：长期支持版本，HTTP Client 标准化。
- Java 17：长期支持版本，密封类、模式匹配等逐步增强。
- Java 21：长期支持版本，虚拟线程正式发布，适合大量阻塞 IO 场景。

追问点：

- 虚拟线程解决什么问题？
- 虚拟线程是不是让 CPU 计算变快？
- 虚拟线程和线程池是什么关系？

回答抓手：

```text
虚拟线程降低了阻塞式并发的线程成本，让一个请求一个线程的模型在 IO 密集场景下更可行。但它不提升 CPU 本身计算能力，也不能替代数据库连接池、限流和背压。
```

## 2. 集合框架高频题

### 2.1 ArrayList 和 LinkedList 区别？

必会回答：

- `ArrayList` 底层是动态数组，随机访问快，尾部追加快，中间插入删除可能需要移动元素。
- `LinkedList` 底层是双向链表，随机访问慢，已定位节点后的插入删除快。
- 实际业务中 `ArrayList` 使用更多，因为 CPU 缓存友好，遍历性能好。

追问点：

- ArrayList 扩容机制？
- 为什么不建议频繁在头部插入 ArrayList？

### 2.2 HashMap 底层原理？

必会回答：

- JDK 8 之后底层是数组 + 链表 + 红黑树。
- 通过 key 的 hash 定位数组下标。
- 哈希冲突时先链表存储，链表过长且数组容量达到阈值后转红黑树。
- 默认负载因子是 0.75，超过阈值会扩容。

必会追问：

- 为什么容量是 2 的幂？
- 为什么负载因子默认是 0.75？
- 链表什么时候转红黑树？
- HashMap 为什么线程不安全？

回答抓手：

```text
容量是 2 的幂时，可以用 (n - 1) & hash 代替取模，效率更高，也有利于散列均匀。
```

### 2.3 ConcurrentHashMap 怎么保证线程安全？

必会回答：

- JDK 7 使用 Segment 分段锁。
- JDK 8 使用 CAS + synchronized + volatile，锁粒度降低到桶级别。
- 读操作多数情况下无锁，依赖 volatile 保证可见性。
- 写入时先 CAS 尝试，冲突再对桶节点加 synchronized。

追问点：

- size 怎么统计？
- put 过程大概是什么？
- 为什么 ConcurrentHashMap 不允许 key/value 为 null？

### 2.4 HashMap、Hashtable、ConcurrentHashMap 区别？

必会回答：

- `HashMap` 非线程安全，允许 null key 和 null value。
- `Hashtable` 线程安全，但方法级 synchronized，粒度粗，性能差，基本不用。
- `ConcurrentHashMap` 线程安全，锁粒度细，适合并发场景。

## 3. Java 并发高频题

### 3.1 线程有哪些状态？

必会回答：

- NEW：新建未启动。
- RUNNABLE：可运行，包括就绪和运行中。
- BLOCKED：等待 synchronized 锁。
- WAITING：无限期等待，如 `Object.wait()`。
- TIMED_WAITING：限时等待，如 `sleep`、带超时的 `wait`。
- TERMINATED：线程结束。

### 3.2 synchronized 和 ReentrantLock 区别？

必会回答：

- `synchronized` 是 JVM 内置锁，语法简单，自动释放。
- `ReentrantLock` 是 JUC 显式锁，需要手动 unlock。
- `ReentrantLock` 支持公平锁、可中断锁、超时获取锁、多个 Condition。
- 普通场景优先用 synchronized，复杂并发控制再用 ReentrantLock。

追问点：

- synchronized 锁升级过程？
- 什么是可重入锁？
- 公平锁和非公平锁区别？

### 3.3 volatile 解决什么问题？

必会回答：

- 保证变量对多线程的可见性。
- 禁止指令重排序。
- 不保证复合操作的原子性。

经典例子：

```java
volatile boolean running = true;
```

常见坑：

```java
volatile int count = 0;
count++; // 不是原子操作
```

追问点：

- volatile 为什么不能保证 i++ 原子性？
- DCL 单例为什么要加 volatile？

### 3.4 ThreadLocal 原理和风险？

必会回答：

- 每个线程内部有一个 `ThreadLocalMap`。
- key 是 ThreadLocal 弱引用，value 是业务对象。
- 适合保存线程上下文，如用户信息、traceId。
- 在线程池中必须 `remove()`，否则可能出现内存泄漏或脏数据复用。

项目表达：

```text
登录鉴权后，可以把当前用户 ID 放到 UserContext 中，请求结束后在过滤器 finally 中清理，避免线程池复用导致串用户。
```

### 3.5 线程池核心参数？

必会回答：

- `corePoolSize`：核心线程数。
- `maximumPoolSize`：最大线程数。
- `keepAliveTime`：非核心线程空闲存活时间。
- `workQueue`：阻塞队列。
- `threadFactory`：线程工厂。
- `handler`：拒绝策略。

执行流程：

```text
先创建核心线程 -> 核心满后进队列 -> 队列满后创建非核心线程 -> 达到最大线程数后触发拒绝策略
```

常见拒绝策略：

- AbortPolicy：抛异常。
- CallerRunsPolicy：调用线程自己执行。
- DiscardPolicy：直接丢弃。
- DiscardOldestPolicy：丢弃队列最老任务。

追问点：

- 线程池参数怎么设置？
- CPU 密集型和 IO 密集型有什么区别？
- 线上线程池打满怎么排查？

### 3.6 AQS 是什么？

必会回答：

- AQS 是 JUC 中很多同步器的基础框架。
- 核心是一个 volatile `state` 和一个 FIFO 等待队列。
- 子类实现获取/释放 state 的规则，AQS 负责排队、阻塞、唤醒。
- ReentrantLock、Semaphore、CountDownLatch 都基于 AQS。

不要死背源码，至少能讲：

```text
抢锁失败的线程会进入 AQS 队列挂起，释放锁时唤醒后继节点继续竞争。
```

## 4. JVM 高频题

### 4.1 JVM 内存结构？

必会回答：

- 程序计数器：记录当前线程执行位置。
- 虚拟机栈：方法调用栈、局部变量表、操作数栈。
- 本地方法栈：Native 方法调用。
- 堆：对象实例主要分配区域，GC 重点管理。
- 方法区/元空间：类元数据、常量、静态变量等。

追问点：

- 哪些区域线程私有？
- 堆和栈有什么区别？
- 什么情况会 StackOverflowError？
- 什么情况会 OutOfMemoryError？

### 4.2 对象创建过程？

必会回答：

```text
类加载检查 -> 分配内存 -> 初始化零值 -> 设置对象头 -> 执行构造方法
```

追问点：

- 对象内存怎么分配？
- TLAB 是什么？
- 对象头里有什么？

### 4.3 类加载过程？

必会回答：

```text
加载 -> 验证 -> 准备 -> 解析 -> 初始化
```

双亲委派模型：

```text
类加载器先委托父加载器加载，父加载器加载不了才自己加载。
```

作用：

- 避免核心类被重复加载。
- 防止用户自定义类篡改 JDK 核心类。

### 4.4 常见垃圾收集器？

必会回答：

- Serial：单线程，适合小内存客户端。
- Parallel：吞吐量优先。
- CMS：老年代低停顿，已逐渐退出主流。
- G1：面向服务端，平衡吞吐和停顿，JDK 9+ 默认。
- ZGC/Shenandoah：低延迟 GC，适合大堆低停顿场景。

初中级重点：

```text
能讲清楚可达性分析、GC Roots、Minor GC、Full GC、STW、G1 的 Region 思路即可。
```

### 4.5 线上 CPU 飙高怎么排查？

回答流程：

```text
top 找到 Java 进程
top -Hp <pid> 找到高 CPU 线程
printf "%x\n" <tid> 转 16 进制
jstack <pid> 查对应 nid 线程栈
结合业务日志定位代码
```

Windows 或本地也可以用：

- JConsole
- VisualVM
- Arthas `thread -n 5`

### 4.6 线上内存溢出怎么排查？

回答流程：

```text
确认 OOM 类型
保留堆转储文件
用 MAT / VisualVM 分析大对象和引用链
检查缓存、集合、ThreadLocal、文件流、连接未释放
修复后压测验证
```

## 5. Spring / Spring Boot 高频题

### 5.1 Spring IoC 是什么？

必会回答：

- IoC 是控制反转，把对象创建和依赖管理交给 Spring 容器。
- 业务代码不再手动 new 依赖，而是通过构造器注入等方式获取 Bean。
- 好处是解耦、便于测试、便于统一管理生命周期。

追问点：

- Bean 生命周期？
- BeanFactory 和 ApplicationContext 区别？
- 循环依赖怎么解决？

### 5.2 Spring AOP 是什么？

必会回答：

- AOP 面向切面编程，用于把日志、事务、权限、限流等横切逻辑从业务中抽离。
- Spring AOP 主要基于动态代理。
- 如果目标类实现接口，通常可用 JDK 动态代理；否则可用 CGLIB。

项目表达：

```text
操作日志、接口耗时统计、权限校验、限流都适合用 AOP 实现，避免每个接口重复写相同代码。
```

### 5.3 Spring 事务什么时候会失效？

高频必会：

- 方法不是 public。
- 同类内部方法自调用。
- 异常被 catch 后没有继续抛出。
- 默认只回滚 RuntimeException 和 Error。
- 数据库引擎不支持事务。
- 没有被 Spring 容器管理。

回答抓手：

```text
@Transactional 本质依赖代理增强，如果绕过代理调用，事务就不会生效。
```

### 5.4 Spring Boot 自动配置原理？

必会回答：

- Spring Boot 根据 classpath 中的依赖和配置条件自动创建 Bean。
- 核心是自动配置类和条件注解，如 `@ConditionalOnClass`、`@ConditionalOnMissingBean`。
- 开发者可以通过配置文件或自定义 Bean 覆盖默认行为。

### 5.5 Spring MVC 请求流程？

必会回答：

```text
请求进入 DispatcherServlet
根据 HandlerMapping 找 Controller
HandlerAdapter 调用目标方法
参数绑定和校验
执行业务逻辑
返回 ModelAndView 或 ResponseBody
消息转换器序列化 JSON
返回响应
```

### 5.6 Spring Security + JWT 怎么讲？

回答结构：

```text
用户登录成功后，服务端签发 JWT。
前端后续请求在 Authorization Header 中携带 Bearer Token。
后端过滤器解析 Token，校验签名和过期时间，构建认证上下文。
退出登录可以把 Token 的 jti 或原 token 放入 Redis 黑名单。
```

追问点：

- JWT 放哪里更安全？
- JWT 如何主动失效？
- Token 被盗怎么办？
- Refresh Token 怎么设计？

## 6. MySQL 高频题

### 6.1 InnoDB 和 MyISAM 区别？

必会回答：

- InnoDB 支持事务、行锁、外键、崩溃恢复，是 MySQL 默认主流引擎。
- MyISAM 不支持事务，表锁为主，适合早期读多写少场景，现在业务系统很少首选。

### 6.2 索引为什么能加速查询？

必会回答：

- MySQL InnoDB 常用 B+Tree 索引。
- B+Tree 高度低，磁盘 IO 少。
- 叶子节点有序，适合范围查询。
- 聚簇索引的叶子节点保存整行数据，二级索引叶子节点保存主键值。

追问点：

- 为什么不用普通二叉树、红黑树、B 树？
- 聚簇索引和非聚簇索引区别？
- 回表是什么？
- 覆盖索引是什么？

### 6.3 索引失效场景？

高频必会：

- 对索引列使用函数或表达式。
- 左模糊 like，例如 `like '%abc'`。
- 隐式类型转换。
- 联合索引不满足最左前缀。
- OR 条件部分字段无索引。
- 数据区分度太低，优化器认为全表扫更划算。

### 6.4 事务 ACID 是什么？

必会回答：

- Atomicity 原子性：事务要么全成功，要么全失败。
- Consistency 一致性：事务前后数据满足约束。
- Isolation 隔离性：并发事务之间互不干扰到一定程度。
- Durability 持久性：提交后数据持久保存。

### 6.5 事务隔离级别？

必会回答：

| 隔离级别 | 可能问题 |
| --- | --- |
| 读未提交 | 脏读、不可重复读、幻读 |
| 读已提交 | 不可重复读、幻读 |
| 可重复读 | 理论仍可能幻读，InnoDB 通过 MVCC 和锁机制缓解 |
| 串行化 | 并发最低，一致性最强 |

MySQL InnoDB 默认是 `REPEATABLE READ`。

追问点：

- 脏读、不可重复读、幻读区别？
- MVCC 是什么？
- undo log、redo log、binlog 各自作用？

### 6.6 MVCC 是什么？

必会回答：

- MVCC 是多版本并发控制。
- 通过隐藏字段、undo log、Read View 实现一致性读。
- 读操作不用阻塞写操作，提高并发能力。

回答抓手：

```text
快照读基于 Read View 判断某个版本对当前事务是否可见；当前读会读取最新数据并可能加锁。
```

### 6.7 一条 SQL 很慢怎么排查？

回答流程：

```text
先看慢 SQL 日志
用 EXPLAIN 分析执行计划
关注 type、key、rows、Extra
检查是否走索引、是否回表、是否 filesort、是否临时表
根据 where/order by/group by 优化索引
必要时改 SQL 或拆分查询
```

## 7. Redis 高频题

### 7.1 Redis 为什么快？

必会回答：

- 基于内存。
- 单线程执行命令，避免线程切换和锁竞争。
- IO 多路复用。
- 数据结构高效。

追问点：

- Redis 真的是单线程吗？
- Redis 6 之后多线程做了什么？

### 7.2 Redis 常见数据结构和场景？

| 数据结构 | 场景 |
| --- | --- |
| String | 缓存、计数器、分布式锁 value |
| Hash | 对象缓存 |
| List | 简单队列、消息列表 |
| Set | 去重、共同关注 |
| ZSet | 排行榜、延迟队列 |
| Bitmap | 签到、活跃用户统计 |
| HyperLogLog | UV 估算 |
| Stream | 消息流、轻量消息队列 |

### 7.3 缓存穿透、击穿、雪崩？

必会回答：

- 穿透：查不存在的数据，缓存和 DB 都没有。解决：缓存空值、布隆过滤器、参数校验。
- 击穿：热点 Key 过期，大量请求打到 DB。解决：互斥锁、逻辑过期、热点不过期。
- 雪崩：大量 Key 同时过期或 Redis 故障。解决：过期时间随机化、多级缓存、限流降级、高可用。

### 7.4 Redis 分布式锁怎么实现？

基础版：

```text
SET lockKey uniqueValue NX EX 30
```

释放锁：

```text
用 Lua 判断 value 是自己的再删除，避免误删别人锁。
```

追问点：

- 为什么 value 要唯一？
- 业务没执行完锁过期怎么办？
- Redisson 看门狗是什么？
- Redis 主从切换时锁安全吗？

### 7.5 Redis Lua 为什么适合秒杀扣库存？

必会回答：

- Redis 执行 Lua 脚本期间具有原子性。
- 可以在一个脚本里完成校验库存、校验是否买过、扣减库存、记录购买标记。
- 避免“先查后扣”之间被其他请求插入导致超卖。

秒杀脚本逻辑：

```text
1. 判断活动是否存在
2. 判断用户是否已经购买
3. 判断库存是否大于 0
4. 扣减库存
5. 记录用户购买标记
6. 返回成功，后续发送 MQ 异步下单
```

## 8. MQ 高频题

### 8.1 为什么引入 MQ？

必会回答：

- 异步削峰：高峰请求先进入消息队列，消费者慢慢处理。
- 系统解耦：下单、通知、积分等模块通过消息解耦。
- 提升响应速度：接口快速返回，耗时任务异步执行。
- 增强可靠性：失败可重试，异常消息进入死信队列。

### 8.2 MQ 会带来什么问题？

必会回答：

- 消息丢失。
- 重复消费。
- 消息积压。
- 顺序问题。
- 分布式事务和最终一致性。

### 8.3 如何保证消息不丢？

回答角度：

```text
生产者确认机制
Broker 持久化
消费者手动 ack
失败重试
死信队列
业务补偿任务
```

### 8.4 如何处理重复消费？

必会回答：

- 消费者必须做幂等。
- 常见方式：唯一索引、业务状态机、去重表、Redis 去重 Key。

秒杀下单例子：

```text
订单表对 activity_id + user_id 建唯一索引。
即使 MQ 重复投递，重复插入也会失败，不会生成两笔订单。
```

### 8.5 死信队列怎么讲？

必会回答：

- 消息多次消费失败、过期或队列拒绝后进入死信队列。
- 死信队列不是垃圾桶，而是异常消息观察和补偿入口。
- 后台可以查看死信原因，修复后重新投递或人工处理。

## 9. 高并发项目必问题

### 9.1 秒杀/抢购系统怎么设计？

回答主线：

```text
活动发布 -> 库存预热到 Redis -> 用户请求限流 -> Lua 原子扣减 -> MQ 异步下单 -> DB 唯一索引防重复 -> 超时未支付取消 -> 库存回滚 -> 对账补偿
```

### 9.2 为什么不能直接扣 MySQL 库存？

回答抓手：

- 高并发下大量请求打到 DB，行锁竞争严重。
- 容易出现超时、连接池耗尽、接口响应慢。
- Redis 适合承接高频库存校验和扣减，DB 作为最终订单和库存结果的持久化。

### 9.3 Redis 扣成功，MQ 发送失败怎么办？

可选方案：

- 本地消息表：同一个事务写订单请求记录，再由定时任务投递 MQ。
- 发送失败立即回滚 Redis 库存和购买标记。
- 记录失败日志，补偿任务扫描恢复。

面试表达：

```text
核心是不能只考虑正常链路，Redis 扣减后任何一步失败都要有补偿路径，否则会出现库存被占用但没有订单的情况。
```

### 9.4 MQ 消费成功，数据库写入失败怎么办？

回答抓手：

- 消费失败不 ack，让 MQ 重试。
- 多次失败进入死信队列。
- 消费者创建订单要幂等。
- 必要时回滚 Redis 库存和购买标记。

### 9.5 超时取消订单怎么做？

常见方案：

- MQ 延迟消息。
- Redis ZSet 延迟队列。
- 定时任务扫描。

回答抓手：

```text
订单创建后是 UNPAID，超过支付期限仍未支付则取消。取消要基于状态机 CAS 更新，只有 UNPAID -> CANCELED 成功时才回滚库存，避免重复取消导致库存多加。
```

### 9.6 接口幂等怎么设计？

常见方式：

- 唯一索引。
- 幂等 Token。
- 业务状态机。
- Redis `SET NX`。
- 请求流水号。

项目表达：

```text
下单接口用 activity_id + user_id 唯一索引防重复订单；支付回调用支付流水号唯一索引防重复处理；取消订单通过状态机保证只回滚一次库存。
```

### 9.7 限流怎么做？

常见算法：

- 固定窗口。
- 滑动窗口。
- 漏桶。
- 令牌桶。

项目表达：

```text
对秒杀接口按 IP、用户、活动维度限流。单机可用 Guava RateLimiter，分布式可用 Redis + Lua 或网关限流。
```

### 9.8 压测报告怎么写？

最小模板：

```text
压测目标：验证抢购接口在高并发下的吞吐和错误率。
环境：CPU、内存、JDK、MySQL、Redis、MQ 配置。
场景：并发用户数、请求总数、库存数量、活动规则。
指标：QPS、平均响应时间、P95、P99、错误率、CPU、内存、DB 连接数。
对比：MySQL 直扣 vs Redis Lua vs Redis Lua + MQ。
结论：瓶颈在哪里，优化后提升多少，还有什么风险。
```

## 10. AI 工程化与时兴面试题

### 10.1 现在 Java 后端为什么要懂 AI？

回答抓手：

```text
AI 已经从单纯聊天变成业务系统的一个能力模块，比如简历分析、客服问答、知识库检索、故障建议、报告生成。后端要负责模型接入、Prompt 管理、文件解析、向量检索、权限隔离、结果校验、降级和成本控制。
```

不要只说“接了大模型 API”，要讲工程问题。

### 10.2 LLM 接入业务系统的基本架构？

常见链路：

```text
用户输入
-> 参数校验和权限校验
-> Prompt 模板组装
-> 调用模型 API
-> 结构化输出解析
-> 业务规则校验
-> 持久化结果
-> 失败降级或重试
```

项目表达：

```text
FixLedger 中 AI 只做票据信息提取和故障建议，核心设备、保修、维修数据仍由后端规则校验后落库，避免模型幻觉直接污染业务数据。
```

### 10.3 什么是 RAG？

必会回答：

- RAG 是 Retrieval-Augmented Generation，检索增强生成。
- 先从知识库中检索相关文档片段，再把片段和用户问题一起交给大模型生成回答。
- 目的是减少幻觉、引入私有知识、让回答可追溯。

基本流程：

```text
文档上传 -> 文档解析 -> 文本切分 -> Embedding 向量化 -> 存入向量库
用户提问 -> 问题向量化 -> 相似度检索 -> 拼接上下文 -> LLM 生成 -> 返回引用来源
```

### 10.4 RAG 有哪些坑？

高频回答：

- 文档切分太粗或太细都会影响效果。
- 向量检索可能召回不准。
- TopK 和相似度阈值需要调参。
- 检索结果需要权限过滤，不能跨用户泄露知识。
- 模型可能忽略上下文产生幻觉。
- 长上下文会增加成本和延迟。

工程兜底：

```text
没有检索到可信内容时，不强行编答案；返回“未找到相关资料”或走人工/规则兜底。
```

### 10.5 Embedding 是什么？

必会回答：

- Embedding 是把文本转换成向量表示。
- 语义相近的文本，向量距离更近。
- 向量库通过相似度搜索找出与问题最相关的文档片段。

追问点：

- 向量维度能不能随便换？
- 换 Embedding 模型后历史向量怎么办？
- 相似度阈值怎么设置？

### 10.6 Prompt 工程在后端怎么管理？

回答抓手：

- Prompt 不要散落在代码里，可以模板化管理。
- 模板要版本化，方便回滚和 A/B 测试。
- 对输出格式要有明确约束，如 JSON Schema。
- 用户输入要做边界处理，防止 Prompt Injection。

### 10.7 什么是结构化输出？

必会回答：

- 要求模型按固定 JSON 结构返回。
- 后端用 DTO / Schema 解析和校验。
- 校验失败可以重试、降级或返回人工处理。

例子：

```json
{
  "deviceName": "空气净化器",
  "brand": "xxx",
  "purchaseDate": "2026-01-01",
  "confidence": 0.86
}
```

工程重点：

```text
模型输出不能直接信任，必须校验日期、枚举、金额、权限和置信度。
```

### 10.8 AI 幻觉怎么处理？

回答抓手：

- 明确限制模型只能基于给定上下文回答。
- RAG 返回引用来源。
- 低置信度结果不自动落库。
- 关键业务使用规则校验。
- AI 失败时有 Mock、规则版或人工兜底。

### 10.9 AI 接口如何做稳定性设计？

必会回答：

- 超时控制。
- 重试和退避。
- 限流。
- 熔断。
- 降级。
- 日志和审计。
- 成本统计。

项目表达：

```text
AI 分析失败不能影响核心业务。简历可以先保存为待分析状态，后续异步重试；设备管理系统中 AI 只是辅助建议，核心 CRUD 不依赖 AI。
```

### 10.10 AI 相关简历怎么写更安全？

推荐写法：

```text
接入 OpenAI-compatible 模型接口，实现简历内容分析/设备故障建议等 AI 辅助能力；通过 Prompt 模板、结构化输出解析、结果校验和规则兜底，避免模型异常影响核心业务流程。
```

不要写：

```text
精通大模型
实现智能 Agent
完全自动生成专业报告
```

除非你真的能讲 Agent 规划、工具调用、记忆、评估、权限和失败恢复。

## 16. 如何用 AI 学而不露馅

AI 可以用，但要换一种用法。

### 不建议的用法

```text
让 AI 直接生成完整项目
让 AI 直接生成简历描述
让 AI 直接生成八股答案然后照背
```

这种方式短期看很快，面试时容易被追问打穿。

### 推荐的用法

```text
1. 让 AI 用通俗语言解释一个概念
2. 自己手写一个最小 Demo
3. 让 AI review 你的代码
4. 让 AI 扮演面试官连续追问
5. 自己整理成 5 句话回答
```

### 每个技术点的验收标准

一个知识点算真正掌握，需要满足：

```text
能画图
能写最小 Demo
能说出项目场景
能说出失败场景
能说出排查方式
```

比如 MQ 不是只知道“异步削峰”，而是要能讲：

```text
为什么下单要异步？
消息发送失败怎么办？
消费者重复消费怎么办？
订单写库失败怎么办？
死信队列怎么处理？
怎么验证 MQ 确实提升了接口响应？
```

### 面试前自测方式

打开录音，随机抽一个题，讲 2 分钟：

```text
HashMap 为什么线程不安全？
Spring 事务为什么会失效？
Redis Lua 为什么能防超卖？
MQ 重复消费怎么处理？
RAG 为什么能减少幻觉？
```

如果你能不用看稿讲完，并且能接住 2 个追问，就可以写进简历或项目讲解。
