## 1. 项目定位

`interview-assistant` 是一个基于 Spring Boot 和 Spring AI 的智能面试助手后端 MVP，当前聚焦两条主链路：

1. 简历上传、解析、存储、AI 分析和规则兜底。
2. 基于简历创建文字模拟面试、答题推进、答案暂存、提前交卷、历史查询和报告生成。

这个项目的正确表达不是“照搬完整平台”，而是：参考成熟面试系统或导师项目的功能拆分思路后，按自己的学习节奏实现一个简化版后端 MVP，并逐步接管其中的文件处理、AI 调用、面试会话状态流转、答案模型和报告链路。

## 2. 项目卡片

项目名：

`interview-assistant` 智能面试助手后端 MVP。

业务背景：

面试准备中，用户往往不知道自己的简历是否突出，也缺少针对简历内容的模拟追问。本项目尝试把“上传简历 -> 分析简历 -> 生成面试题 -> 作答 -> 查看报告”做成一条后端闭环。

用户是谁：

准备 Java 后端面试、需要简历诊断和文字模拟面试练习的求职者。

核心流程：

1. 用户上传简历文件。
2. 后端校验文件类型和大小。
3. 基于文件内容计算 hash，命中重复简历时直接返回已有记录。
4. 使用 Apache Tika 解析文本并做基础清洗。
5. 将原始文件上传到 RustFS / S3 兼容对象存储。
6. 保存简历元信息和解析文本到 PostgreSQL。
7. 调用 Spring AI 进行结构化简历分析，失败时回退到本地规则分析。
8. 用户基于某份简历创建文字模拟面试会话。
9. 系统根据简历关键词生成规则版题目，并保存题目快照。
10. 用户暂存或提交答案，提交后推进 `currentQuestionIndex`。
11. 面试完成后生成规则版报告。

技术栈：

- Java 21
- Spring Boot 4.0.1
- Spring Web MVC
- Spring Data JPA
- PostgreSQL
- Spring AI 2.x
- Apache Tika
- RustFS / S3 Compatible Storage
- Maven

我真实参与/接管的部分：

- 重点接管简历上传解析链路、内容 hash 去重、AI 分析兜底、面试会话状态流转和答案模型演进。
- 当前能讲清 `ResumeUploadService`、`ResumeGradingService`、`InterviewSessionService` 的主流程和过渡设计。
- 后续准备亲手补测试、修正报告读取答案表的问题、完善历史详情和报告聚合逻辑。

最能讲的 2 个难点：

1. AI 简历分析的工程兜底：模型调用可能失败、输出可能不符合结构化格式，所以项目使用 DTO + 输出转换器接收结构化结果，并在异常时回退到规则版评分，保证上传主流程不会完全不可用。
2. 面试会话状态推进：`currentQuestionIndex` 表示下一道待答题的索引，而不是最近一道已答题索引；暂存答案不推进索引，提交答案才推进；完成后禁止继续提交或暂存。

## 5. 1 分钟项目话术

`interview-assistant` 是一个基于 Spring Boot 和 Spring AI 的智能面试助手后端项目，目前只做后端 MVP，不急着铺完整平台能力。它的主流程是：用户上传简历，后端先做文件校验和内容 hash 去重，再用 Apache Tika 解析简历文本，把原文件存到 S3 兼容对象存储，同时把简历元信息和文本保存到 PostgreSQL。之后系统会调用大模型做结构化简历分析，如果 AI 调用失败或输出异常，就回退到规则版评分，保证主流程可用。

第二条链路是文字模拟面试：用户基于简历创建会话，系统根据简历关键词生成规则版题目，用户可以暂存答案或提交答案，提交后推进到下一题，完成后生成规则版报告。这个项目有参考成熟系统的功能拆分思路，但简历和面试里我只讲 `interview-assistant` 本身。我当前准备接管的重点是面试答案表、报告聚合逻辑、核心流程测试，以及后续的 AI 出题和评估。

## 6. 3 分钟项目话术

`interview-assistant` 的定位是智能面试助手后端 MVP，目标不是一开始做一个完整商业平台，而是先把“简历诊断 + 文字模拟面试”这两条最核心链路跑通。它适合作为我接管 AI 辅助项目的主项目，因为业务边界比较清楚，能同时覆盖 Java 后端常见能力和 AI 工程化能力。

第一条链路是简历模块。用户上传简历后，Controller 只负责接收文件，核心编排在 `ResumeUploadService`。服务层先调用文件校验，限制非法类型和大小；然后用文件内容计算 hash，而不是依赖文件名，这样同一份简历改名后也能识别重复；如果 hash 命中已有记录，就直接返回已有简历，避免重复存储。没有重复时，系统用 Apache Tika 解析简历文本并做清洗，再把原始文件上传到 RustFS / S3 兼容存储，最后把文件元信息、hash、文本内容保存到 PostgreSQL。

简历分析部分接入了 Spring AI。项目通过 Prompt 模板和结构化 DTO 要求模型返回可解析结果，再持久化到简历分析表。但这里我不会把 AI 当成完全可靠的业务源，所以代码里保留了规则版兜底：如果 AI 调用失败、输出解析失败或简历文本异常，系统会生成本地规则评分和建议，至少保证用户能得到可解释结果。这一块面试时可以引出 AI 服务超时、结构化输出校验、降级策略和异步化演进。

第二条链路是文字模拟面试。用户基于简历创建会话，系统会优先复用同一简历最近一条未完成会话，避免重复创建多场并行面试。当前题目是规则版生成，主要根据简历里的 Spring Boot、MySQL、Redis、Docker 等关键词生成。会话表里保存题目快照和当前进度，其中 `currentQuestionIndex` 的语义是“下一道待答题的索引”，这能让获取当前题、提交答案和判断完成状态比较清楚。暂存答案只保存内容，不推进索引；提交答案才推进索引；索引达到题目总数后，会话进入 `COMPLETED`，此后不允许继续提交。

我现在接管时看到的一个重点是答案模型正在从过渡方案演进到独立答案表。早期可以把题目和答案都放在 `questionsJson` 里，方便快速跑通，但后续要做历史详情、单题评分、AI 评估和 PDF 导出时，答案必须独立建表。最新代码里已经有 `InterviewAnswerEntity`，暂存和提交也开始写答案表；下一步需要检查报告和历史详情是否真正从答案表聚合数据，并补单元测试防止状态推进和报告统计回归。

所以这个项目我会诚实表述为：AI 辅助开发后，我正在按真实代码重新审查、复盘、补测试和接管关键模块。能写进简历的部分，只写我能画流程、能讲代码、能回答失败场景并能亲手修改的内容。

## 7. 项目关联的面试题方向

文件上传：

- 文件类型校验只看后缀够不够？
- 大文件上传怎么处理？
- 对象存储失败时数据库要不要回滚？

AI 工程化：

- AI 分析失败怎么办？
- 模型输出 JSON 解析失败怎么办？
- 如何防止 Prompt Injection？
- 为什么 AI 结果不能直接写核心业务字段？

数据库和事务：

- 简历 hash 去重只靠代码判断够不够？
- 为什么还需要数据库唯一约束？
- 上传文件、对象存储和数据库保存如何处理一致性？

面试会话：

- 为什么复用未完成会话？
- 暂存和提交有什么区别？
- 当前题索引如何避免越界和重复提交？

模型演进：

- 为什么先用 JSON 快照，后面再拆独立答案表？
- 独立答案表如何保证同一会话同一题只有一条答案？
- 报告统计应该从哪里读答案？
